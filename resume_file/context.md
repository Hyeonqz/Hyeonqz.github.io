## 3. 이력서 전략

### 핵심 원칙
- PDF 1장 = 숫자 성과 bullet만 (리뷰어 30초 판단용)
- 노션 포트폴리오 = 기술 깊이, 의사결정 과정 상세 서술
- PDF에 노션 링크 포함 → "궁금하면 들어와서 봐라"

### PDF bullet 작성 패턴
```
[문제] + [기술 선택] + [숫자 결과]
예시: "Kafka Request-Reply 아키텍처 설계 — 타임아웃 50%→0%, CS 60% 감소"
```

### PDF 경력 bullet (확정본)
```
(주) 큐뱅 | Server Developer | 2024.01 - 현재

· Kafka Request-Reply 아키텍처 설계 — 분산 환경 Reply 라우팅 이슈
  해결, 타임아웃 50%→0%, 수동 CS 월 5건→2건 (60% 감소)

· Redis Cache-Aside + 트랜잭션 경계 최적화 — Polling 120회 기준
  DB 쿼리 360회→0회, 응답시간 200ms→30ms (85% 개선)

· Final API 패턴 도입으로 POS↔큐뱅 결제 상태 동기화 구현
  — CS 월 10건→0~1건 (95% 감소), 연간 40시간 수동 공수 절감

· Outbox 패턴 + SFTP 세션 풀링으로 파일 유실률 100%→0% 달성
  — 장애 감지 수동(불특정) → 30분 이내 자동 감지 및 재처리
```

### 아까운 요소 (현재 이력서에서 묻혀있는 것)
- `es-hangul Java 포팅 → Maven Central 배포` — Activities에 묻혀있음
- 토스 자사 오픈소스 포팅이라는 맥락이 강력한 어필 포인트
- 오픈소스 기능 추가 완료 후 별도 섹션으로 격상 예정

## 4. 노션 포트폴리오 전략

### 구조
```
메인 페이지
├── About (PDF 링크 + 한줄 소개)
├── 프로젝트 4개 (각 상세 페이지)
└── 오픈소스 (es-hangul, 기능 추가 후 작성)
```

### 프로젝트 배치 순서 (임팩트 기준)
1. Kafka 통합 결제 아키텍처 (분산 시스템 설계 깊이)
2. 거래 상태 조회 Redis 개선 (숫자 임팩트 가장 명확)
3. CS 95% 감소 Final API (비즈니스 임팩트 + 의사결정)
4. Outbox 패턴 파일유실 0% (안정성 + 패턴 적용)

### 각 페이지 내부 구조
```
1. 배경 & 문제 (텍스트)
2. AS-IS 다이어그램 (Mermaid)
3. 의사결정 근거 — 옵션 비교 테이블
4. TO-BE 다이어그램 (Mermaid)
5. 트러블슈팅
6. 결과 & 회고 (숫자 + 남은 과제)
```

### 설계 사고력 드러내는 서술 원칙
- 결과가 아닌 **"왜 이 선택을 했는지"** 중심
- 옵션 비교 테이블 포함 (트레이드오프 명시)
- **인지된 한계** 솔직하게 서술 → 오히려 신뢰감 상승
- 회고에 남은 과제 포함
