2. 🧭 두 Kafka 항목 — 확정 스토리 (혼동 금지)

① Kafka 실시간 Push — 단방향 알림 전달 문제

배경(=설계 동기): TCP Channel이 특정 인스턴스에만 존재 → API 서버가 POS로 직접 Push 불가 ("누가 쏠 것인가")
해결: produce/consume 분리 + Redis 소유 서버 판별 + 전략 패턴
결과: API↔TCP 완전 디커플링 · 중복 Push 0건

② Kafka Request-Reply — 팀 간 장애 전파 격리 문제

배경(=설계 동기): 국내·해외 결제 시스템 통합 시 동기 REST 직접 연결은 상대 시스템 장애·지연이 그대로 전파 + 엔드포인트·방화벽·인증 결합 → 기존 Kafka 인프라 재사용, Request-Reply로 가용성 디커플링
문제(=트러블슈팅): 분산 환경 Reply 토픽 Consumer 인스턴스 불일치 → 요청 서버가 응답 미수신, 타임아웃
해결: 인스턴스별 전용 Reply 토픽 + CorrelationId 매칭키, 실패 건 DLQ 적재 후 DB 저장
결과: 팀 간 장애 전파 격리 · Reply 타임아웃 0건 · DLQ→DB로 운영팀 거래 모니터링
⚠️ 규칙: ①은 "채널 소유권", ②는 "장애 전파 격리" — 서로 다른 문제. DRAFT 작성 시 두 항목의 배경이 겹치면 안 됨 (겹치면 같은 문제를 두 번 쓴 것으로 읽힘).

3. 🎤 면접 대비 노트 (이력서에는 넣지 않음)

Q. "다시 설계한다면 Kafka Request-Reply를 또 쓰겠는가?"

"목표는 그대로 디커플링 — 달라지는 건 그걸 사는 방법. 당시엔 Kafka가 가용성 디커플링을 구조적으로 공짜로 줬지만, 비동기로 동기를 흉내 내느라 CorrelationId 매칭·타임아웃 관리·응답 유실 재처리라는 운영 복잡도를 대가로 치렀다. 다시 한다면 역할을 나눈다: 본질이 동기인 결제 요청-응답은 gRPC(타입 계약) + 서킷브레이커·엄격한 타임아웃으로 장애 전파 방어, 본질이 비동기인 것(결과 통지·정산 이벤트·후처리)은 Kafka에 유지. 교훈: 기술의 본질을 거슬러 쓰면 원래 문제는 풀려도, 새 문제를 만들어 푸는 비용이 든다."

→ 트레이드오프 인지 + 재설계 관점을 보여주는 답변. 포트폴리오(Notion) "결과 & 회고" 섹션에 활용 가능.

4. ✍️ DRAFT 작성 규칙 (이번 개편에서 확립)

  1. 배경 = 설계 동기, 문제 = 구현 중 트러블슈팅 — 둘을 섞지 않는다 (Request-Reply 사례).
  2. 사업 성과는 "기여"가 아니라 "처리/사용/기반"으로 — 매출 80% 기여 ❌ → 거래망이 80% 처리 ⭕, 가맹점 확보 ❌ → 1,000개+가 사용 ⭕.
  3. 0회·0건 주장에는 조건 단서 — "캐시 미스 시에만 접근", "실패 건 자동 재처리로" 등 검산 방어.
  4. % 뒤에는 절대값 근거 — 50% 단축 ❌ → 12주→6주 (50%↓) ⭕. 근거 없는 %는 기간·건수로 대체 (테스트 비용 사례).
  5. 표기 통일: (95%↓) 형식 · — 대시 · 가운뎃점 나열 · 진행 중 프로젝트는 현재형.
  6. 문서 간 수치 100% 일치 — 이력서/경력기술서/포트폴리오가 다른 숫자를 말하면 그 순간 신뢰 붕괴.

6. 🧩 <li> 안에 하위 <li> 넣는 법 (중첩 리스트)

핵심 규칙: HTML에서 <li> 바로 안에 <li>를 넣는 건 불가합니다. 반드시 부모 <li> 안에 <ul>을 하나 더 넣고, 그 <ul> 안에 자식 <li>를 넣습니다. 이 프로젝트엔 이미 class="sub-list"가 있어(Kafka·Redis 항목에서 사용 중) 그대로 재사용하면 스타일도 맞습니다.

❌ 잘못된 구조 (li 안에 li 직접)

<li>간편결제사 7개사 연동
    <li>라인페이</li>   <!-- ✗ 브라우저가 자동 교정, 들여쓰기 깨짐 -->
</li>

✅ 올바른 구조 (li 안에 ul → 그 안에 li)

<li>
    국내·국외 간편결제사 <strong>7개사</strong> 연동 개발 — 연동 결제망이 전사 거래액의 80% 처리
    <ul class="sub-list">
        <li>국내: 하나은행 등</li>
        <li>해외: 라인페이 · 알리페이 · 위챗페이 · 유니온페이</li>
    </ul>
</li>

포인트: 부모 텍스트를 먼저 쓰고, 닫는 </li> 직전에 <ul class="sub-list">…</ul>을 넣습니다. 자식 <li>는 그 <ul> 안에만 존재합니다.

💡 판단 포인트: 이 프로젝트에서 sub-list는 지금까지 "배경/문제/해결/결과" 같은 설계 서사에만 썼습니다. 결제사 나열처럼 단순 목록에도 sub-list를 쓰면 시각적으로 "이것도 트러블슈팅 항목인가?"하는 혼동을 줄 수 있습니다.
→ 나열이 목적이면 괄호 안에 인라인(현행: "라인·알리·위챗·유니온·하나은행 등")이 더 깔끔할 수 있고, 국내/해외 구분처럼 계층 정보가 있을 때만 중첩 리스트가 값어치를 합니다. 어떤 목적인지에 따라 선택하세요.
