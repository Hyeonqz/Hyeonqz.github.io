## 2. 두 Kafka 항목 — 확정 스토리 (배경 절대 겹치지 말 것)

**① Kafka 실시간 Push — 단방향 알림 전달 문제 (채널 소유권)**
- 배경(=설계 동기): TCP Channel이 특정 인스턴스에만 존재 → API 서버가 POS로 직접 Push 불가 ("누가 쏠 것인가")
- 진행/해결: produce/consume 분리 + Redis 소유 서버 판별 + 전략 패턴
- 결과: API↔TCP 완전 디커플링 · 중복 Push 0건

**② Kafka Request-Reply — 팀 간 장애 전파 격리 문제 (가용성 디커플링)**
- 배경(=설계 동기): 국내·해외 결제 시스템 통합 시 동기 REST 직접 연결은 상대 시스템 장애·지연이 그대로 전파 + 엔드포인트·방화벽·인증 결합 → 기존 Kafka 인프라 재사용, Request-Reply로 통신 통합
- 문제(=트러블슈팅): 분산 환경 Reply 토픽 Consumer 인스턴스 불일치 → 요청 서버가 응답 미수신, 타임아웃
- 해결: 인스턴스별 전용 Reply 토픽 + CorrelationId 매칭키, 실패 건 DLQ 적재 후 DB 저장
- 결과: 팀 간 장애 전파 격리 · Reply 타임아웃 0건 · DLQ→DB로 운영팀 거래 모니터링

> ⚠️ ①은 "채널 소유권", ②는 "장애 전파 격리". 배경이 겹치면 같은 문제를 두 번 쓴 것으로 읽힘.

4. **% 뒤에는 절대값 근거** — 50% 단축 ❌ → 12주→6주(50%↓) ⭕. 근거 없는 %는 기간·건수로 대체.

7. **긴 불릿은 구조(sub-list)가 아니라 단어를 빼서 압축** — sub-list는 "배경/문제/해결/결과" 서사 전용. 2줄 wrap은 정상.

## 7. 면접 노트 (이력서엔 넣지 않음 · 포트폴리오 회고용)

**Q. 다시 설계한다면 Kafka Request-Reply를 또 쓰겠는가?**
목표는 그대로 디커플링 — 달라지는 건 방법. 당시 Kafka가 가용성 디커플링을 구조적으로 줬지만, 비동기로 동기를 흉내 내느라 CorrelationId 매칭·타임아웃 관리·응답 유실 재처리라는 운영 복잡도를 대가로 치렀다. 다시 한다면 역할 분리: 본질이 동기인 결제 요청-응답은 gRPC(타입 계약)+서킷브레이커·엄격한 타임아웃, 본질이 비동기인 것(결과 통지·정산·후처리)은 Kafka 유지. 교훈: 기술의 본질을 거스르면 원래 문제는 풀려도 새 문제를 만들어 푸는 비용이 든다.
