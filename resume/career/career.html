기술적 개선 사항

분산 환경 실시간 Kafka Push 아키텍처 설계 — Redis 기반 채널 소유권 문제 해결

배경 결제 완료 결과를 POS로 실시간 통지하는 Push 기능이 필요했고, 서버 증설(다중 인스턴스) 과정에서 Netty Channel이 특정 서버에만 존재해 인스턴스 간 채널 소유권 문제 발생 — 통지 누락·중복 위험
해결 API 서버는 Kafka produce만 담당, Socket 서버 전체가 인스턴스별 고유 groupId로 consume 후 Redis 기반 소유 서버 판별로 Channel 보유 서버만 Push 실행
결과 API 서버 ↔ TCP 서버 완전 디커플링   중복 Push 0건   장애 격리 구조 확보   [상세보기]

Kafka Request-Reply 아키텍처 설계

배경국내·해외 결제 시스템 통합 시 동기 REST 직접 연결은 상대 시스템의 장애·지연이 그대로 전파되고 엔드포인트·방화벽·인증 결합 발생 → 기존 Kafka 인프라를 재사용해 Request-Reply 패턴으로 통신 통합 (가용성 디커플링)
문제분산 환경에서 Reply 토픽 Consumer 인스턴스 불일치로 요청 서버가 응답을 수신하지 못해 타임아웃 빈번 발생
해결인스턴스별 전용 Reply 토픽 구독 + CorrelationId 매칭키로 요청-응답 상관관계 보장, 실패 건 DLQ 적재 후 DB 저장
결과팀 간 장애 전파 격리 달성 · Reply 타임아웃 0건 · DLQ 적재 건 DB 저장으로 운영팀 거래 모니터링 활용 [상세보기]

AWS Bedrock LLM 기반 운영 자동화 — 가맹점 영문명 의역 배치

배경 신규 가맹점 접수 시 한글명을 영문으로 직역 생성하여 가독성 저하 → 운영 담당자가 B/O에서 월 약 100건을 수동 수정하며, 담당자 부재 시 가맹점 접수 누락·지연 발생
해결AWS Bedrock(Claude) 연동 일 1회 배치 설계·구현
  • english_name_status 플래그(DRAFT / AI_TRANSLATED / MANUAL / FAILED)로 처리 상태 관리
  • original_english_name 컬럼으로 원본 보존(롤백·감사 대응)
  • 실패 건 스킵 후 다음 배치 재처리 또는 수동 API 호출을 통한 처리
예시직역 결과 KORAILUTONG_HWAJANGPUMJUM → 의역 결과 KORAILRETAIL_COSMETICS 형태로 파스칼케이스 + 카테고리 구분 영문명 자동 생성
결과수동 공수: 월 100건 → 5건 (95%↓)   영문명 품질 일관화   담당자 부재 리스크 제거

거래량 급증 대비 DB 병목 선제 해소 — Redis Cache-Aside + 트랜잭션 경계 최적화

배경거래량 급증을 감당할 시스템이 필요해 Pinpoint APM으로 병목을 선제 분석 — 결제 상태 Polling 최대 120회 × 회당 조인 쿼리 2회·업데이트 1회, 거래당 최대 360쿼리가 RDB로 직행해 거래 증가가 DB 부하를 증폭시키는 구조를 최우선 병목으로 판단
해결Cache-Aside 패턴 도입, 트랜잭션 커밋 후 캐시 갱신 순서 보장으로 더티 리드 방지
결과상태 조회 거래당 최대 120회(RDB 쿼리 360회) → 0회 (Redis 최우선 조회, 쓰기는 상태 변경 시에만)  API 응답 200ms→30ms (85%↓, Pinpoint APM 측정) — 거래량이 늘어도 DB 부하가 늘지 않는 구조 확보   [상세보기]

가맹 접수 파일 유실 0건 — Outbox 패턴 + SFTP 세션 풀링

배경가맹 접수 시 수신한 파일을 비동기로 처리하는 과정에서 실패 시 유실 감지 불가 — 실패 건은 전량 유실되어 수동 복구에 의존
해결Outbox 패턴으로 파일 처리 상태 DB 영속화 → 실패 건 자동 재처리, SFTP 세션 풀링으로 연결 안정성 확보, 파일 I/O 비동기(@Async) 처리
결과파일 유실 0건 (기존: 실패 시 전량 유실·수동 복구)  응답시간 4s→1s (75%↓)  도입 후 10,000건 무장애  장애 감지 수동(불특정) → 30분 내 자동 감지  [상세보기]

모놀리식 애플리케이션 → 멀티 모듈 및 DDD 기반 이벤트 주도 아키텍처(EDA) 전환

배경 단일 모놀리식에서 공통 코드가 비대화돼 거의 모든 변경이 공통 영역을 건드림 → merge 충돌·테스트 상호 간섭·배포-수정 반복으로 팀 병렬 작업이 막힘. 특정 기능의 장애가 전체 서비스로 전파되고, 모듈 간 동기 결합은 장애 전파·향후 MSA 분리 시 병목 리스크로 이어짐.
해결 1. Gradle 멀티 모듈 분리(서버 아키텍처 기준 API/TCP 경계 정렬) → 빌드·테스트·소유권 독립
2. 패키지 레벨 헥사고날 아키텍처로 도메인을 인프라에서 격리(포트/어댑터) → 기술 교체 유연성 확보
3. ApplicationEventPublisher + @TransactionalEventListener(AFTER_COMMIT)로 모듈 간 동기 트랜잭션 결합 제거
결과 모듈 경계 독립화: 빌드·테스트·소유권 분리로 팀 병렬 작업 가능 (부수적으로 Gradle 빌드 55s→25s, 55%↓) [상세보기]
트랜잭션 결합 제거: 모듈 간 동기 호출을 AFTER_COMMIT 이벤트로 전환 → 한 모듈 트랜잭션이 타 모듈로 전파되지 않음
도메인-인프라 격리: 포트/어댑터로 기술 교체 유연성·테스트 용이성 확보
